Class

자바에서 클래스(class)란 객체를 정의하는 틀 또는 설계도와 같은 의미로 사용됩니다.
자바에서는 이러한 설계도인 클래스를 가지고, 여러 객체를 생성하여 사용하게 됩니다.
클래스는 객체의 상태를 나타내는 필드(field)와 객체의 행동을 나타내는 메소드(method)로 구성됩니다.
즉, 필드(field)란 클래스에 포함된 변수(variable)를 의미합니다.
또한, 메소드(method)란 어떠한 특정 작업을 수행하기 위한 명령문의 집합이라 할 수 있습니다.
  • Class 명의 첫글자는 대문자, Class 명이 여러 단어로 구성될 경우, 각 단어별로 대문자 : GamePlayer
  • Class 명의 정의는 명사로,
  • 자바에서 제공되는 Library, API의 클래스명은 사용하지 말 것

Static는 객체 참조형과 달리 Heap에 등록되지 않고 고정 메모리로 설정되기 때문에 GC의 제어를 받지 않습니다.
즉 Static 객체에 생성된 데이터 또는 Object에 만약 계속적인 추가가 발생한다면 이는 메모리 Leak의 원인이 될 수 있습니다.
  • 상수(Constant)를 제공하기 위해서
  • 업무 프로세스 상에서, 각각의 업무 클래스를 공통적으로 유지해야할 Object가 존재할 경우
  • 클래스 Instance를 만들지 않기 위한 대안을 제공하기 위해서
  • Utility 함수로 제공하기 위해서

Static에 대한 이야기를 하면 Singleton에 대한 언급하지 않고 넘어 갈 수 없다.
Singleton은 오직 하나의 객체만 생성되는 것을 보장하는 패턴으로 static 을 이용해서 Singleton Pattern을 많이 만들게 된다. 
                          
class SingletonClass
{
  private static SingletonClass instance;
  
  // 해당 클래스의 객채 생성을 막을 수 있는 방법 
  private SingletonClass()
  {

  }

  public static SingletonClass getInstance()
  {
    if ( instance == null)
      instance = new SingletonClass();
    return instance;
  }
}

Method에 대한 Final 처리
Spaceship 클래스가 Base Class( 부모 클래스 )가 되어 하위 클래스에 상속될 때, 하위 클래스에서 getVelocity 라는 메소드는 재정의 될 수 없다. 
  
public class SpaceShip
{
  public final int getVelocity()
  {
    return this.velocity;
  }
}
Class에 대한 Final 처리
Final Class는 Base Class(부모 클래스)로 부터 상속 받을 수 없다.
public final class BaseBall { }
변수에 대한 Final 처리
Final Variables의 경우에는 해당 값이 선언된 이후 변경할 수 없다.
Final은 반드시 초기화되어야 하며, Class를 이용한 생성자 생성 시점에 Final을 통한 변수 초기화가 가능하다.

  • 밀접하게 연관된 클래스(Class)사이에서 코드를 공유하고자 할 때,
  • non-static, non-final 값으로 Class를 설계하고자 할 때,
  • 많은 공통적인 함수들을 사용할 필요가 있는 클래스들에 대한 설계가 필요하거나, 접근제어자에 의한 클래스 설계가 필요할 때 ( public 보다는 proctected 나 private 을 활용해야하는 경우 )

Java의 Inner Class 혹은 Nested Class는 하나의 클래스로, 클래스나 인터페이스 내부에 선언합니다.
  • Inner Class의 장점은 Outer Class에 있는 모든 자원에 접근 가능함
  • Outer$Inner.class
  • Inner Class 독단적으로 사용할 수 없는 단점이 잇음 
  
class Outer
{
  int number = 1000;
    
  class Inner
  {
    int number = 2000;
    
    public void getValue()
    {
      int number = 3000;
      
      System.out.println(number); //1000
      System.out.println(this.number); //2000
      System.out.println(Outer.this.number); //3000
    }
  }
}

public class InnerClassExam
{
  public static void main(String[] args)
  {
    Inner i = new Outer().new Inner();
    i.getValue();
  }
}

try catch 문에 대한 고민
Framework 및 Library에서 공통 Exception을 제공할 때, try catch 문을 사용하면 프로세스 상의 심각한 문제를 초래할 수 있다.
에외 처리는 상당히 깊이 있게 다루어져야할 문제이다.
단순히 예외를 try catch 문으로 전체로 감싸버린다면 이는 서버, Application의 심각한 문제를 초래할 수 있다.
try catch 문에 대해서 일어날 지도 모르는 문제에 대해서 대응하기 보다 해당 프로세스에 대한 명확한 정의에 따른 프로그램 설계가 더욱 중요할 수 있다.
