Rules - Class, Inteface

잘 설계된 모듈과 그렇지 못한 모듈을 구별 짓는 가장 중요한 속성 하나는 모듈 내부의 데이터를 비롯한 구현 세부 사항을 다른 모듈에 잘 감추느냐의 여부다. 잘 설계된 모듈은 구현 세부 사항을 전부 API 뒤쪽에 감춘다.
  • 정보은닉
    정보 은틱은 여러가지 이유로 중요한데, 그 대부분은 정보 은닉이 시스템을 구성하는 모듈 사이의 의존성을 낮춰서, 각자 개별적으로 개발하고, 시험하고, 최적화하고, 이해하고, 변경할 수 있도록 한다는 사실에 기초한다. 그렇게 되면 시스템의 개발속도가 올라가는 데, 각각의 모듈을 병렬적으로 개발할 수 있기 때문이다.
    정보은닉 원칙이 좋은 성능을 자동적으로 보장하는 것은 아니지만, 효과적인 성능 튜닝을 가능하게 하는 것은 사실이다.
    시스템이 완성된 다음에 어떤 모듈이 성능 문제를 일으키는지 프로파일링 하기 용이하기 때문이다.
    정보 은닉 원칙은 소프트웨어 재사용 가능성을 높인다.
    모듈 간의 의존성이 낮으므로 각 모듈은 다른 소프트웨어 개발에도 유용하게 쓰일 수 있다.
    정보은닉 은닉 원칙은 대규모 시스템의 개발 과정의 위험성도 낮춘다.
    "각 클래스와 멤버는 가능한 한 접근 불가능하도록 만들라는 것"
    : 개발 중인 소프트웨어의 정상적인 동작을 보장하는 한도 내에서 가장 낮은 접근 권한을 설정하라는 것 
                                    
package STUDY.Java.ClassSameple.PackagePrivateSample;

// 외부에서 사용불가한 class로 선언됨 - Package Private한 Class 임 
class PackagePrivateClass {

}
    만일 package-private 로 선언된 최상위 레벨 클래스나 인터페이스를 사용하는 클래스, 즉 사용자 클래스가 하나뿐이라면, 해당 최상위 클래스를 사용자 클래스의 private 중척 클래스로 만들 것을 고려해보기 바란다. 그러면 패키지 전체가 아니라 단 하나의 클래스 만이 해당 클래스의 접근 권한을 가지게 된다. 하지만 최상위 레벨 package-private 클래스의 접근 권한을 낮추는 것보다 public 클래스의 접근 권한을 낮추는 것이 더 중요하다.
    • private
    • package-private
    • protected
    • public
  • 캡슐화
객체 필드는 절대로 public 으로 선언하면 안된다.
: 비-final 필드나 변경 가능 객체에 대한 final 참조 필드를 public으로 선언하면, 필드에 저장될 값을 제한할 수 없게 된다.
이는 메서들 통하지 않고도 필드의 값을 변경할 수 있기 때문이다.
변경 가능한 public 필드를 가진 클래스는 다중 스레드에 안전하지 않다.
길이가 0이 아닌 배력은 언제나 변경가능하므로, public static final 배열 필드를 두거나, 배열 필드를 반환하는 접근자를 정의하면 안된다. 
                             
private static final Thing[] PRIVATE_VALUES = { ... };
public static final List<Things> VALUES = Collections.unmodifiable(Arrays.asList(PRIVATE_VALUES));

private static final Thing[] PRIVATE_VALUES = { ... };
public static final Thing[] values() {
  return PRIVATE_VALUES.clone();
}
  • 접근 권한을 가능한 낮추라
  • 최소한의 public API를 설계한 다음, 다른 모든 클래스, 인터페이스, 멤버는 API에서 제외하라
  • public static final 필드를 제외한 어떤 필드도 public 필드로 선언하지 마라.
  • public static final 필드가 참고하는 객체는 변경 불가능 객체로 만들어라.

때때로, 필드를 모으는 것 말고는 아무 기능도 없는 쓰레기 클래스를 만들고 싶어질 때가 있을 것이다. 
                            
// 이런 저급한 클래스는 절대로 public으로 선언하지 말 것 
class point {
  public double x;
  public double y;
}
이런 클래스는 데이터 필드를 직접 조작할 수 있어서 캡슐화의 이점을 누릴 수가 없다. API를 변경하지 않고서는 내부 표현을 변경할 수도 없고, 불변식도 강제할 수 없고, 필드를 사용하는 순간에 어떤 동작이 실행되도록 만들 수 도 없다.
위의 코드를 최적화 했을 경우 
                            
// 접근자 메서드와 수정자를 이용한 데이터 캡슐화 
class Point {
  private double x;
  private double y;

  public Point(double x, double y){
    this.x = x;
    this.y = y;
  }

  public double getX() { return x; };
  public double getY() { return y; };

  public void setX(double x){ this.x = x; };
  public void setY(double y){ this.y = y; };

}
선언된 패키지 밖에서도 사용 가능한 클래스에는 접근자 메서드를 제공하라.
그래야 클래스 내부 표현을 자유로이 수정할 수 있게 된다. public 클래스의 데이터 필드를 공개하게 되면, 그 내부 표현을 변경할 수 있게 된다. 변경하면 이미 작성된 클라이언트 코드를 깨뜨리게 되기 때문이다.
