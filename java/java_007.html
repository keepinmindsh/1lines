Rules - Exception

// 예외를 끔직하게 남용한 사례 
                            
// Logic 상에 발생할 수 있는 에러를 막기위해서 아래와 같이 사용했다면 이는 예외를 잘못 사용한 것이다.                               
try{
  int i = 0;
  while(true){
    range[i++].climb();
  }
}catch(ArrayIndexOutOfBoundsException e){

}
이름이 말하듯이, 예외는 예외적인 상황에만 사용해야한다. 평상시 제어 흐름에 이용해서는 안 된다.
잘 설계된 API는 클라이언트에서 평상시에 제어 흠의 일부로 예외를 사용하도록 강요해서는 안된다.

  • 점검 지정 예외(checked exception)
  • 실행 시점 예외(runtime exception)
  • 오류(error)
호출자(caller) 측에서 복구할 것으로 여겨지는 상황에 대해서는 점검지정 예외를 이용해야 한다는 것
프로그래밍 오류를 표현할 때는 실행시점 예외를 사용하라. 대부분의 실행시점 예외는 선행조건 위반을 나타낸다.즉, 클라이언트가 API명세에 기술된 규약을 지키지 않았다는 뜻이다. 

                           
// 예외를 점검하도록 지정된 메서드 호출
try{
  obj.action(args);
}catch(TheCheckedException e){
  // 예외적인 상황 처리
  ...
}

// 상태 검사 메서드를 거쳐서 무점검 예외 메서드 호출
if(obj.actionPermiited(args)){
  obj.action(args);
}else{
  // 예외적인 상황 처리
  ...
}

숙련도가 낮은 프로그래머와 달리, 프로그래밍 전문가는 코드 재사용을 위해 노력할 뿐만 아니라, 보통 높은 수준을 코드를 재사용한다. 코드를 재사용하면 좋다는 일반규칙은 예외에도 어김없이 적용된다.

메서드가 하는 일과 뚜련한 관련성이 없는 예외가 메서드에서 발생하면 당혹스럽다. 추상화 수준이 낮은 곳에서 발생한 예외를 그대로 밖으로 전달하면 이런 일이 생긴다. 당혹스러울 뿐더라, 추상화 수준이 높은 API가 구현 세부 사항으로 오염되는 일까지 번진다. 다음번 릴리스에 상위 계층의 구현이 바뀌면 해당 계층에서 밸상하는 예외도 바뀔 것이고, 결국 클라이언트 프로그램이 깨질 가능성이 높아진다. 
                            
// 예외변환
try {
  // 낮은 수준의 추상화 계층 이용
  ...
}catch(LowerLevelException e){
  throw new HigherLevelException(...);
}

// 예외 연결
try{
  ... // 낮은 수준의 추상화 계층에 이용
}catch(LowerLevelException cause){
  throw new HigherLevelException(cause);
}

// 예외연결 지원 생성자를 갖춘 예외
class HigherLevelException extends Exception {
  HigherLevelException(Throwable cuase){
    super(cause);
  }
}
대부분의 표준 예외들은 예외 연결 지원 생성자를 구비하고 있다. 그런 생성자가 없는 예외는 Throwable.initCause 메서드를 호출하면 하위 계증예외를 연결할 수 있다. 예외 연결 기능을 사용하면 프로그램 안에서 예외의 원인에 접근할 수 있을 뿐 아니라, 최초에 발생한 예외의 스택 추적 정보를 상위 계정 예외에 통합할 수 있게 된다.

점검지정 예외는 독립적으로 선언하고, 해당 예외가 발생하는 상황은 JavaDoc @throws 태그를 사용하여 정확하게 밝혀라.
메서드를 올바르게 사용하려면, 메서드에 던져지는 예외에 대한 설명이 문서에 있어야 한다.
Javadoc @throws 태그를 사용해서 메서드에서 발생가능한 모든 무점검 예외에 대한 문서를 남겨라.
하지만 메서드 선언부의 throws 뒤에 무점검 예외를 나열하진 마라.
같은 이유로 동일한 예외를 던지는 메서드가 많다면, 메서드마다 문서를 만드는 대신, 해당 예외에 대한 문선느 클래스의 문서화 주석에 남겨도 된다.

무점검 예외 때문에 프로그램이 죽으면, 시스템은 자동적으로 해당 예외의 스택 추적 정보를 출력한다.
오류 정보를 포착해 내기 위해서는, 오류의 상세 메세지에 "에외에 관계된" 모든 인자와 필드의 값을 포함 시켜야 한다.
관련된 모든 "실제 데이터(hard data)"를 담는 것은 중요한 반면, 중언부언 떠드는 것은 별로 도움되지 않는다.
오류를 적절히 포착하는 정볼를 상세 메세지에 담는 한가지 방법은, 상세한 정보를 요구하는 생성자를 만드느 것이다. 
                            
/**
 * Constuct an IndexOutofBoundsException.
 *
 * @param lowerBound the lowest legal index value
 * @param upperBound the highest legal index value plus one
 * @param index the actual index value.
 *
 */
 public IndexOutOfBoundsException(int lowerBound, int upperBound, int index){
   // 오류를 포착하는 상세 메세지 생성
   super("Lower bound : " + lowerBound +
         ", Uppper bound : " + upperBound +
         ", Index : " + index 
   )

   // 프로그램에서 이용할 수 있도록 오류 정보 보관
   this.lowerBound = lowerBound;
   this.upperBound = upperBound;
   this.index = index;
 }

예외를 던지고 난 뒤에도 객체의 상태가 잘 정의된, 사용 가능한 상태로 남아있으면 좋다.
설사 어떤 연산을 수행하는 도중에 문제가 생겼다 해도 말이다. 호출자 측에서 오류 상태를 복구할 수 있는 점검 지정 예외의 경우에는 특히 더 그렇다.
변경 가능한 객체의 경우에는 실제 연산을 수행하기 전에 인자의 유효성을 검사하는 것이 가낭 보편적인 방법이다. 
                          
public Object pop(){
  if ( size == 0 )
    throw new EmptyStactException();
    Object result = element[--size];
    element[size] = null; // 만기 참조 제거 
    return result;
}
메서드 명세에 포함된 예외가 발생하더라도 객체 상태는 메서드 호출 이전과 동일하게 유지되어야 한다는 것이다.

API 설계자라 예외를 발생 할 수있다고 선언했다면, 그것은 API를 이용할 프로그래머에게 무언가 알리려는 것이다. 무시하지마라!
호출 대상 메서드를 빈 Catch 블록이 붙은 try 문으로 감싸면, 예외를 쉽게 무시할 수 있다. 
                          
// catch 블록을 비워놓으면 예외는 무시된다. - 심히 의심스러운 코드 
try{
  ...
}catch(SomeException e){

}
빈 Catch 블록은 예외를 선언한 목적, 그러니까 예외적인 상황을 반드시 처리하도록 강제한다는 목적에 배치된다. 적어도 Catch 블록안에는 예외를 무시해도 괜찮은 이유라도 주석으로 남겨두어야 한다.
빈 Catch 블록으로 예외를 무시하는 프로그램은 오류가 생겨도 조용히 실행을 계속한다. 그러다 어느 순간, 오류와는 아무 상관도 없는 지점에서 불시에 죽어버린다.
